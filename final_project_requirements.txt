  <h1>JavaFX Game Development — Final Project Requirements</h1>

  <h2>Objective</h2>
  <p>
    Develop a complete game using <strong>Java</strong> and <strong>JavaFX</strong> that demonstrates mastery of the
    concepts and skills learned throughout the course. This project is intended to evaluate not only whether your team
    can produce a working game, but also whether you can design it carefully, organize it properly, explain it clearly,
    and justify the technical decisions you made along the way.
  </p>
  <p>
    Because the use of Generative AI is unrestricted for this assignment, the grading emphasis is not based solely on
    how many features you were able to produce. Instead, this project also evaluates your understanding of your own
    code, your ability to apply course concepts correctly, and your ability to explain how your system works.
  </p>

  <div class="highlight">
    <strong>Generative AI Policy:</strong><br>
    The use of Generative AI is <strong>completely unrestricted</strong> for this assignment. However, any material
    obtained from Generative AI, Internet sources, or class materials must be integrated thoughtfully and cited
    academically where relevant. You are responsible for understanding, correcting, and defending all code and design
    decisions included in your submission, regardless of where they came from.
  </div>

  <h2>Project Requirements</h2>

  <h3>Proposal Submission</h3>
  <p>
    Before beginning implementation, each team must submit a written proposal for instructor approval. The purpose of
    the proposal is not simply to list features. Instead, the proposal must function as a <strong>design plan</strong>
    for the game your team intends to build.
  </p>
  <ul>
    <li>
      <strong>File Format:</strong> The proposal must be submitted as a plain text <code>.txt</code> file.
    </li>
    <li>
      <strong>Required Identification:</strong> The proposal must include the team name and the names of all team members.
    </li>
    <li>
      <strong>Submission Method:</strong> The proposal must be emailed to the professor before the proposal due date
      listed in the final project dates.
    </li>
    <li>
      <strong>Approval:</strong> The instructor will review each proposal for scope, feasibility, and technical fit with
      the course requirements. A proposal may require revision before it is approved.
    </li>
  </ul>

  <h4>Required Proposal Contents</h4>
  <ul>
    <li>
      <strong>Game Concept:</strong> Provide a clear narrative description of the game. Explain what kind of game it is,
      what the player does, what the main goal is, and what makes the game interesting.
    </li>
    <li>
      <strong>Core Gameplay Loop:</strong> Explain the repeating cycle of play. In other words, describe what the player
      is expected to do again and again during gameplay, and explain how those actions lead toward success or failure.
    </li>
    <li>
      <strong>Win and Loss Conditions:</strong> Clearly describe how the game is won, how it is lost, and how the game
      changes when one of those conditions is reached.
    </li>
    <li>
      <strong>MVC Design Plan:</strong> Explain how the project will be divided using the
      <strong>Model–View–Controller (MVC)</strong> design pattern.
      <ul>
        <li><strong>Model:</strong> Describe the data, rules, and game information that belong in the model.</li>
        <li><strong>View:</strong> Describe the screens, layouts, and visual elements that belong in the view.</li>
        <li><strong>Controller:</strong> Describe how input will be handled and how user actions will affect the model.</li>
      </ul>
    </li>
    <li>
      <strong>Lecture Coverage Plan:</strong> Explain how your team intends to demonstrate at least one feature from
      each of the 11 class lectures. It is understood that, when this project is first released, not all 11 lectures
      will yet have been taught. However, the lecture titles in the syllabus should be sufficient to help you plan at a
      high level, and you may ask the instructor for clarification if needed.
    </li>
    <li>
      <strong>Risk and Simplification Plan:</strong> Identify which parts of the game are most difficult and explain what
      features would be reduced or removed first if time becomes limited. This is meant to demonstrate planning and
      prioritization.
    </li>
  </ul>

  <h3>Contribution and Presentation</h3>
  <ul>
    <li>All team members must actively contribute to the project.</li>
    <li>All team members must present part of the final presentation.</li>
    <li>
      Each team member must be able to explain the code, design, and implementation choices associated with the portion
      of the project they present.
    </li>

    </li>

  <h3>Game Project Requirements</h3>
  <ul>
    <li>
      The project must be written in <strong>Java</strong> and use <strong>JavaFX</strong> for the user interface.
      No other Java UI toolkit may be used.
    </li>
    <li>
      External libraries and external code packages are <strong>not permitted</strong>, except for course-provided
      materials explicitly allowed by the instructor.
    </li>
    <li>
      Your program may <strong>not assume Internet access</strong> during execution.
    </li>
    <li>
      The game must be runnable on both <strong>Windows</strong> and <strong>Unix-based systems</strong> such as
      <strong>Linux</strong> and <strong>macOS</strong>.
    </li>
    <li>
      If the game includes networked play, the same <code>main</code> method must be used for both client mode and
      server mode. In other words, your game must have only one project and one <code>main</code> method, and
      command-line arguments must be used to determine whether that one program instance runs as a client or as a server.
    </li>
    <li>
      The opening interface of the game must be defined in an <code>.fxml</code> file.
    </li>
    <li>
      The root node of that <code>.fxml</code> file must be a <code>BorderPane</code>.
    </li>
    <li>
      The <code>top</code> region of the <code>BorderPane</code> must contain a <code>MenuBar</code>.
    </li>
    <li>
      The <code>MenuBar</code> must contain at least three <code>Menu</code> controls labeled
      <strong>File</strong>, <strong>Edit</strong>, and <strong>Help</strong>.
    </li>
    <li>
      The <strong>File</strong> menu must contain at least one <code>MenuItem</code> labeled <strong>Exit</strong>.
    </li>
    <li>
      The <strong>Edit</strong> menu must contain at least one <code>MenuItem</code> labeled <strong>Preferences</strong>.
    </li>
    <li>
      The game must provide at least <strong>three user preferences</strong> that the player can change.
      These preferences must be <strong>persistent</strong>, meaning they are saved when the program exits and are
      loaded again the next time the program runs.
    </li>
    <li>
      The <strong>Help</strong> menu must contain at least three <code>MenuItem</code> controls labeled
      <strong>Controls</strong>, <strong>Rules</strong>, and <strong>About</strong>.
    </li>
    <li>
      The <strong>Controls</strong> screen must explain what inputs the player can use and what actions those inputs perform.
    </li>
    <li>
      The <strong>Rules</strong> screen must explain the game itself, including the objective and the relevant rules of play.
    </li>
    <li>
      The <strong>About</strong> screen must include the following items, in any order:
      <ul>
        <li>team name</li>
        <li><code>CI245 Java Programming</code></li>
        <li><code>Final Project</code></li>
        <li>names of all team members</li>
        <li><code>Spring 2026</code></li>
        <li>name of the game</li>
      </ul>
    </li>
    <li>
      The design and layout of the <strong>Controls</strong>, <strong>Rules</strong>, and <strong>About</strong>
      screens are left to the students.
    </li>
    <li>
      The game’s code and features must demonstrate at least one feature from each of the 11 class lectures.
    </li>
    <li>
      Any code taken from Generative AI, Internet sources, or class materials must be cited academically in the source code.
    </li>
    <li>
      All forms must inherit from <code>JavaFXWindow</code>.
    </li>
    <li>
      The project must use the <strong>MVC design pattern</strong>.
    </li>
  </ul>

  <div class="highlight">
    <strong>MVC Requirement:</strong><br>
    Model–View–Controller (MVC) is required for this project. In this pattern, the
    <strong>Model</strong> contains the game’s data, rules, and game information;
    the <strong>View</strong> is responsible for visual presentation using JavaFX;
    and the <strong>Controller</strong> handles user input and translates it into actions on the model.
    Your design should keep gameplay logic out of the JavaFX scene graph whenever possible.
  </div>

  <h4>Definition of Game Information</h4>
  <p>
    In this assignment, the phrase <strong>game information</strong> refers to the important facts that describe the
    current condition of the game at a given moment. Examples include player positions, scores, health values, remaining
    lives, inventory contents, enemy locations, whether the game is paused, whose turn it is, whether the player has won,
    and any other values that determine what is happening in the game. This information belongs in the model, not in the
    visual nodes that happen to display it.
  </p>

  <h4>What the View Means in a JavaFX Project</h4>
  <p>
    In some JavaFX projects, the view may be very small and may consist mostly of an <code>.fxml</code> file together
    with the JavaFX controls it defines. In a simple project, that is acceptable. In larger projects, however, teams may
    also create additional Java classes whose job is primarily visual presentation. For example, a team might create a
    class to build a game board, render a heads-up display, manage a custom drawing surface, update visible labels, or
    refresh the screen from model data. Such classes are still part of the view as long as their purpose is presentation
    rather than game decision-making.
  </p>
  <p>
    It is also normal in JavaFX for an <code>.fxml</code> file to directly reference a controller class. That does not
    mean the <code>.fxml</code> file itself is the controller. Rather, the <code>.fxml</code> file defines the layout,
    while the associated controller class receives events and coordinates interactions. Some projects may need only FXML
    plus controller classes for their view layer, while other projects may benefit from additional helper classes devoted
    to rendering or display updates.
  </p>

  <h4>MVC Expectations</h4>
  <ul>
    <li>Keep the model independent of JavaFX UI classes whenever practical.</li>
    <li>Place game rules, victory conditions, defeat conditions, scoring, and game information in the model.</li>
    <li>
      Use the view to define, construct, and update what the player sees. This can include FXML layout files, JavaFX
      nodes, scene construction code, and helper classes whose purpose is display.
    </li>
    <li>
      Use controller classes to translate keyboard, mouse, menu, or other user actions into operations on the model.
    </li>
    <li>
      Avoid placing large amounts of gameplay logic directly inside small event-handling methods such as methods attached
      to button clicks, key presses, or mouse events. Even when those methods appear inside a controller class, they
      should usually delegate to clearer controller helper methods or model methods rather than trying to contain all of
      the game rules themselves.
    </li>
    <li>
      View-related code may update what is shown on the screen based on model data, but it should not be the place where
      core game rules are decided.
    </li>
    <li>
      Keep screens organized clearly, such as menu screens, gameplay screens, pause screens, help screens, and game-over screens.
    </li>
    <li>
      Treat animation and rendering as the visual presentation of game information, not as the source of truth for the
      game itself.
    </li>
    <li>
      Design the project so that major gameplay logic could be tested or reasoned about without needing to launch the full JavaFX interface.
    </li>
  </ul>

  <h3>Documentation and Coding Standards</h3>
  <ul>
    <li>Full JavaDoc commenting is mandatory.</li>
    <li>All course coding standards must be followed.</li>
    <li>Code should be clear, well-organized, well-documented, and easy to understand.</li>
    <li>
      Use <code>@author</code> tags in each class to list contributors in order of contribution.
    </li>
    <li>
      FXML files should include author comments in the same order of contribution when appropriate.
    </li>
    <li>
      Classes, methods, and variables should use correct Java naming conventions.
    </li>
    <li>
      Your package structure, class structure, and overall project organization should reflect deliberate design rather
      than a collection of unrelated files.
    </li>
  </ul>

  <h3>Final Written Report</h3>
  <p>
    In addition to the code and presentation, each team must submit a final written report. This report is an important
    part of the project because it documents how the finished game satisfies the course requirements and how the team
    approached the design and implementation of the system.
  </p>
  <ul>
    <li>
      <strong>File Format:</strong> The final written report must be submitted as a plain text <code>.txt</code> file.
    </li>
    <li>
      <strong>Approximate Length:</strong> The report should be approximately <strong>4 to 6 pages of plain text</strong>.
      As a rough guideline, this will usually be around <strong>1,500 to 2,500 words</strong>, depending on formatting.
    </li>
  </ul>

  <h4>Required Report Contents</h4>
  <ul>
    <li>
      <strong>Project Summary:</strong> Briefly describe the completed game and its major features.
    </li>
    <li>
      <strong>MVC Justification:</strong> Explain how your project implements the MVC design pattern. Identify what
      belongs to the model, what belongs to the view, and what belongs to the controller.
    </li>
    <li>
      <strong>Lecture Requirement Coverage:</strong> Include a separate section that explains how the project satisfied
      each of the 11 lecture requirements. It is your responsibility to identify and justify these.
    </li>
    <li>
      <strong>Proposal vs. Final Project Comparison:</strong> Explain how the final game differed from the original
      proposal, if it did at all. If features were added, removed, simplified, or significantly redesigned, explain what
      changed and why. Part of the report grade will be based on how clearly and honestly these deviations are explained.
    </li>
    <li>
      <strong>AI and Source Usage Disclosure:</strong> Explain what was written directly by team members, what was
      AI-assisted, and what came from other sources. Explain what changes or corrections were required.
    </li>
    <li>
      <strong>Technical Reflection:</strong> Explain what parts of the design worked well, what caused difficulty, and
      what you would redesign if you built the project again.
    </li>
    <li>
      <strong>Deep Technical Explanation:</strong> Choose at least one major subsystem and explain it in depth. Examples
      include the game loop, collision handling, input handling, save/load behavior, enemy behavior, or management of
      game information.
    </li>
  </ul>
